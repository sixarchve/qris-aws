QRIS Payment System Flow
========================

This report describes the current repository architecture and runtime flow.


1. Runtime Components
---------------------

- Nginx Reverse Proxy:
  - Path: nginx/
  - Default port: 80
  - Responsibilities: Unified entry point for the stack. Routes `/merchant/` and `/customer/` requests to the respective frontend apps, and `/api/`, `/ws`, `/` requests to the Go backend API.
  - Dependency: Configured to start immediately (`service_started`) when the backend starts, ensuring health checks can always be accessed.

- Backend API:
  - Path: backend/
  - Language/framework: Go + Gin
  - Default port: 8080 (internal / proxied)
  - Responsibilities: Config loading, host auto-resolution inside containers, database connection retrying, QRIS generation, transaction creation, payment confirmation, WebSocket broadcasting.

- Merchant dashboard:
  - Path: frontend/
  - Framework: React + Vite (configured with base `/merchant/`)
  - Default port: 5173 (internal / proxied)
  - Responsibilities: select merchant, generate dynamic QRIS payload.

- Customer app:
  - Path: customer-app/
  - Framework: React + Vite (configured with base `/customer/`)
  - Default port: 5174 (internal / proxied)
  - Responsibilities: scan QRIS, create transaction, confirm payment, poll transaction status.

- PostgreSQL:
  - Source of truth for merchants and transactions. Supported with SSL mode (`DB_SSLMODE`).

- Redis:
  - Speed/caching layer for merchant and transaction data. Supported with AUTH password and TLS connection options.

- S3 Receipt Store:
  - Store for uploading JSON receipt files directly to an AWS S3 bucket.

- WebSocket hub:
  - Path: backend/internal/websocket/
  - Endpoint: /ws?merchant_id=<merchant_uuid>
  - Responsibilities: keep merchant dashboard connections, queue a small in-memory backlog when a merchant is disconnected, and flush pending notifications when that merchant reconnects.


2. Startup Flow
---------------

Backend startup begins in `backend/cmd/main.go`.

Flow:
- Load configuration from environment, fallback to repo-root `.env`.
- Auto-resolve `localhost`/`127.0.0.1` settings to Docker hostnames (`db` / `redis`) if running inside a container.
- Connect to PostgreSQL using a retry loop (5 attempts, 2-second sleep).
- Create `pgcrypto` extension if needed.
- Run GORM AutoMigrate for merchants and transactions.
- Seed default active merchants:
  - TEST001 / Kantin FILKOM UB
  - TEST002 / TESTING STORE
- Connect to Redis.
- Warm Redis merchant cache with all active merchants.
- Create usecases and handlers.
- Initialize the WebSocket hub.
- Start HTTP server on port 8080.
- Start Nginx Reverse Proxy server on port 80 to route traffic.
- On SIGINT or SIGTERM, gracefully stop HTTP server and websocket connections.


3. Merchant List Flow
---------------------

Endpoint:
GET /api/merchants

Flow:
- Merchant dashboard requests active merchants.
- Backend queries PostgreSQL for merchants where `is_active = true`.
- Backend returns the merchant list.
- Dashboard uses the merchant UUID as `merchant_id` for QRIS generation.


4. Generate QRIS Flow
---------------------

Endpoint:
GET /api/qris?merchant_id=<merchant_uuid>&amount=<amount>

Flow:
- Frontend submits merchant UUID and amount.
- Backend validates amount is a positive integer.
- Backend validates merchant_id is a UUID.
- Backend loads merchant from PostgreSQL.
- Backend caches merchant in Redis with key `merchant:<qr_id>`.
- Backend starts speculative prefetch for related active merchants.
- Backend generates a QRIS payload with:
  - tag 00: payload format indicator
  - tag 01: dynamic QR indicator
  - tag 26.00: QRIS domain
  - tag 26.01: merchant QRID, for example TEST001
  - tag 52: merchant category code
  - tag 53: IDR currency code
  - tag 54: amount
  - tag 58: country ID
  - tag 59: merchant name
  - tag 60: city, MALANG
  - tag 63: CRC checksum
- Backend returns `qris_payload`, `merchant_id`, and `amount`.


5. Customer Scan Flow
---------------------

Endpoint:
POST /api/transactions/scan

Request:
```json
{
  "qr_payload": "<qris_payload>",
  "merchant_id": "TEST001",
  "amount": 1000
}
```

Flow:
- Customer app scans QRIS payload.
- Customer app extracts merchant QRID from tag 26.01.
- Customer app extracts amount from tag 54.
- Customer app submits `qr_payload`, `merchant_id`, and `amount`.
- Backend accepts `merchant_id` as either merchant UUID or QRID.
- For QRID input, backend checks Redis merchant cache first.
- If Redis misses, backend queries PostgreSQL and caches the merchant.
- Backend parses the QRIS payload server-side.
- Backend validates QR CRC.
- Backend validates QR merchant QRID matches selected merchant.
- Backend validates QR amount matches request amount.
- Backend creates a PENDING transaction in PostgreSQL.
- Backend stores transaction JSON in Redis (key: `transaction:<transaction_id>`, TTL: 10 minutes).
- Backend returns created transaction data.


6. Transaction Status Flow
--------------------------

Endpoint:
GET /api/transactions/:id

Flow:
- Backend validates transaction ID is a UUID.
- Backend checks Redis key `transaction:<transaction_id>`.
- If Redis hits:
  - Backend unmarshals transaction JSON and returns it.
- If Redis misses:
  - Backend queries PostgreSQL, stores fresh transaction JSON in Redis, and returns it.
- If cached data is corrupted:
  - Backend deletes the corrupted cache key and falls back to PostgreSQL.


7. Payment Confirmation Flow
----------------------------

Endpoint:
POST /api/transactions/:id/confirm

Flow:
- Backend validates transaction ID is a UUID.
- Backend updates status to `SUCCESS` in PostgreSQL.
- Backend invalidates Redis cache key `transaction:<transaction_id>`.
- Backend saves the JSON receipt file to S3ReceiptStore (backed up by LocalReceiptStore in a composite layout).
- Backend triggers an asynchronous goroutine to publish the notification payload to the local WebSocket hub.
- WebSocket hub pushes the `transaction_notification` message to all connected merchant dashboard instances for that merchant.
- Dashboard displays a transaction banner with merchant, amount, and status details.


8. Redis Usage
--------------

Redis is a speed/caching layer. PostgreSQL remains the source of truth.

Merchant cache:
- Key format: `merchant:<qr_id>`
- TTL: 30 minutes
- Used for QRID-based lookup during customer scan.
- Warmed on backend startup.

Transaction cache:
- Key format: `transaction:<transaction_id>`
- TTL: 10 minutes
- Used for repeated status polling.

Failure behavior:
- If Redis is down, cache reads miss and cache writes are skipped.
- Backend continues through PostgreSQL.


9. Identifiers
--------------

Merchant ID:
- UUID primary key in PostgreSQL.
- Used by the merchant dashboard when calling GET `/api/qris`.

Merchant QRID:
- Human-readable QR merchant code stored in `qr_id`.
- Example: TEST001.
- Embedded inside QRIS payload tag 26.01.
- Used by the customer app after scanning QRIS.

Transaction ID:
- UUID primary key in PostgreSQL.
- Used by status and confirmation routes.
